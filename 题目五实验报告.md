## 摘要

本项目基于原生 UDP Socket，在应用层实现可靠传输的底层协议骨架。系统通过自定义 Header 在 UDP Payload 中封装分组序号、发送时间戳和固定长度数据；发送端维护未确认队列，并通过独立 Timer 线程执行 RTO 定时重传；接收端根据分组序号返回累计 ACK；发送端 ACK 线程收到确认后滑动发送窗口，并根据被确认分组的 Timestamp 计算 RTT/SRTT。同时，系统实现了连续 3 个重复 ACK 触发快速重传，以及接收端乱序缓存处理。在此基础上，发送端新增 Q-Learning 智能拥塞控制器，以 1 个 RTT 为控制周期，统计 RTT 趋势和丢包事件，构造 6 状态 Q-Table，并在每个周期结束时选择 CWND 保持、加 1 或减半。

## 1. 实验目标

本实验聚焦题目五中的可靠传输底层架构，目标如下：

1. 在 UDP 之上自定义应用层数据包 Header。
2. 使用序号和累计 ACK 实现基础确认机制。
3. 发送端维护未确认队列，支持 RTO 超时重传。
4. 使用多线程拆分发送、ACK 接收和定时器扫描逻辑。
5. 设计 Q-Learning 智能拥塞控制器，根据网络反馈动态调整发送窗口。
6. 支持固定窗口与 Q-Learning 控制模式对比，便于观察智能控制效果。

## 2. 系统模块

项目保留以下核心文件：

| 文件 | 作用 |
| --- | --- |
| `protocol.py` | 数据包与 ACK 的封装、解析 |
| `sender.py` | 可靠发送端，负责滑动窗口、ACK 处理、重传和 AIMD/Q-Learning 控制 |
| `receiver.py` | 接收端，负责收包、累计确认和乱序缓存 |

## 3. 协议格式

### 3.1 数据包格式

| 字段 | 大小 | 含义 |
| --- | --- | --- |
| Sequence Number | 4 Bytes | 分组序号 |
| Timestamp | 8 Bytes | 发送端写入的 Unix 时间戳 |
| Payload | 1024 Bytes | 固定长度数据负载 |

在代码中，数据包格式由 `protocol.py` 中的常量定义：

```python
PAYLOAD_SIZE = 1024
DATA_HEADER_FORMAT = "!Id"
ACK_FORMAT = "!i"
```

其中 `!` 表示网络字节序，`I` 表示 4 字节无符号整数，`d` 表示 8 字节浮点数时间戳。

### 3.2 ACK 格式

ACK 报文只包含一个 4 字节字段：

| 字段 | 大小 | 含义 |
| --- | --- | --- |
| ACK Number | 4 Bytes | 当前累计确认到的最大连续序号，`-1` 表示尚无连续分组 |

## 4. Header 封装

`protocol.py` 提供四个核心函数：

- `pack_data_packet(seq, timestamp, payload)`：把序号、时间戳和负载封装为 UDP 可发送的字节串。
- `unpack_data_packet(packet)`：从收到的 UDP 数据中解析序号、时间戳和负载。
- `pack_ack(ack_number)`：封装 ACK。
- `unpack_ack(packet)`：解析 ACK。

发送端每次发送新分组或重传分组时，都会重新调用 `pack_data_packet()`，确保 Header 中的时间戳对应本次发送时间。

## 5. 可靠传输机制

### 5.1 发送端未确认队列

发送端使用 `unacked` 字典保存尚未收到 ACK 的分组。每个分组记录：

- `payload`：原始负载，用于重传。
- `last_send_monotonic`：最近一次发送时刻，用于计算是否超时。
- `wire_timestamp`：写入 Header 的 Unix 时间戳。
- `transmissions`：发送次数。

### 5.2 滑动窗口

发送端主线程按当前拥塞控制窗口控制在途分组数量。固定窗口模式下该值不变；Q-Learning 模式下该值由控制器在每个 RTT 周期结束时更新：

```text
while next_seq < end_seq and len(unacked) < window_size:
    send_new_packet(next_seq)
```

这样可以避免一次性把所有 UDP 数据报发出，也为智能拥塞控制器提供直接的调节入口。

### 5.3 累计 ACK

接收端维护 `expected_seq`：

- 如果收到的 `seq == expected_seq`，说明分组按序到达，接收端推进 `expected_seq`。
- 如果收到更大的序号，说明出现乱序，接收端先缓存该序号。
- 每次处理完数据后，接收端返回 `expected_seq - 1` 作为累计 ACK。

发送端收到 ACK 后，会删除所有 `seq <= ack_number` 的未确认分组。

## 6. 定时重传

发送端启动独立 Timer 线程，每隔一小段时间扫描未确认队列：

```text
if now - last_send_monotonic >= RTO:
    retransmit(packet)
```

当检测到超时时，发送端会：

1. 使用原 Payload 重新封包。
2. 写入新的发送时间戳。
3. 更新最近发送时间。
4. 增加发送次数。
5. 重新通过 UDP Socket 发送。

由于 UDP 本身不保证可靠性，这一机制保证了丢失的分组最终可以再次发送。

## 7. 快速重传与乱序处理

基础 RTO 重传必须等待超时，恢复速度较慢。发送端 ACK 线程会统计重复累计 ACK：

```text
if duplicate_ack_count >= 3:
    retransmit(ack_number + 1)
```

当接收端先收到后续分组、但缺少当前期望序号时，会持续返回相同的累计 ACK。发送端连续收到 3 个重复 ACK 后，立即认为 `ack_number + 1` 对应分组可能丢失，并执行快速重传。

接收端使用 `buffered` 集合缓存乱序到达的序号。缺失分组到达后，接收端会连续推进 `expected_seq`，从而一次性确认已经缓存的后续分组。

## 8. Q-Learning 智能拥塞控制器设计

Q-Learning 控制器由 `sender.py` 内置实现，发送端通过 `--cc-mode qlearning` 启用。控制器不直接修改协议格式，而是每隔 1 个 RTT 统计本周期网络反馈，再通过 Q-Table 决定下一周期的 CWND。

### 8.1 状态设计

控制器严格按照题目要求，将过去 1 个 RTT 内的网络特征离散为 6 个状态：

| 状态维度 | 取值 | 含义 |
| --- | --- | --- |
| RTT 趋势 | `rtt_up` / `rtt_down` / `rtt_stable` | 本周期平均 RTT 相对上一周期变大、变小或平稳 |
| 丢包事件 | `loss` / `no_loss` | 本周期是否发生 RTO 或快速重传 |

因此 Q-Table 的状态集合为 `3 × 2 = 6` 个，例如 `rtt_up|loss` 表示 RTT 上升且本周期出现丢包，通常说明窗口过大或网络排队严重。

### 8.2 动作设计

动作集合为：

| 动作 | 窗口变化 |
| --- | --- |
| `0` | CWND 保持不变 |
| `1` | CWND + 1，主动探测更多带宽 |
| `2` | CWND / 2，用于响应拥塞信号 |

控制器不是每个 ACK 都立即调窗，而是在一个 RTT 周期结束时，根据本周期统计得到的状态，通过 epsilon-greedy 策略选择下一周期动作。

### 8.3 奖励函数

奖励函数采用复合形式，综合考虑吞吐量、RTT、重传和超时四个维度：

```text
R = throughput_weight × acked_packets
    - timeout_weight × timeout_events
    - retx_weight × retransmissions
    - rtt_weight × max(0, avg_rtt_ms - target_rtt_ms)
```

其中：
- `acked_packets`：本周期成功确认的分组数
- `avg_rtt_ms`：本周期平均 RTT（ms）
- `target_rtt_ms`：目标 RTT，仅惩罚超出目标的部分
- `retransmissions`：重传次数
- `timeout_events`：超时事件数

训练采用两阶段奖励：正常链路阶段使用 `throughput=1.4`、`timeout=14`、`retx=2.5`、`rtt=0.006`、`target_rtt=40ms`；带宽骤降阶段使用 `throughput=1.2`、`timeout=18`、`retx=3.5`、`rtt=0.008`、`target_rtt=50ms`。第二阶段提高超时和重传惩罚，使模型更重视突发拥塞后的恢复能力。

### 8.4 Q 值更新

每个 RTT 周期结束时，控制器使用标准 Q-Learning / Bellman 公式更新上一个状态动作对：

```text
Q(s, a) = Q(s, a) + alpha * (reward + gamma * max Q(s', a') - Q(s, a))
```

程序参数 `--q-alpha` 控制学习率，`--q-gamma` 控制未来奖励折扣，`--epsilon` 控制探索概率。使用 `--qtable-file artifacts/models/active/q_table.json` 时，程序会在结束时保存当前使用的 Q 表，下一次运行可以继续加载已有学习结果。历史权重备份保存在 `artifacts/models/backups/`。由于单次传输不一定收敛，项目提供 `train_q_learning.py` 进行多轮循环训练，并可通过 `--epsilon-decay` 逐轮降低探索概率。

需要说明的是，Q-Learning 的 CWND 原始曲线不一定像固定规则控制器一样平滑。训练阶段存在 epsilon-greedy 随机探索，6 个离散状态会把不同网络情况映射到同一状态，动作也只有保持、加一和减半三类，因此窗口变化可能呈现抖动或阶梯状跳变。绘图时可以保留原始曲线，同时叠加滑动平均曲线，用原始曲线展示真实反馈，用平均曲线观察整体趋势。

### 8.5 多轮训练方法与训练结果

为获得收敛的 Q 表，项目实现了多轮迭代训练框架 `train_q_learning.py`。每轮训练启动独立的发送端和接收端进程，发送端使用当前 Q 表进行 epsilon-greedy 学习，轮次结束后根据评分指标选出最佳 Q 表。训练完成后使用 `--eval-rounds` 指定的轮次进行纯贪心（epsilon=0）评估。

**训练参数配置（q_table_mixed_v4）：**

| 参数 | 值 | 说明 |
| --- | --- | --- |
| 训练轮次 | 60 | 正常链路 30 轮 + 带宽减半 30 轮，每轮 300 分组 |
| 学习率 α | 0.005 | 控制 Q 值更新步长 |
| 折扣因子 γ | 0.90 | 未来奖励权重 |
| 初始 ε | 0.06 / 0.08 | 两阶段起始探索概率 |
| ε 衰减 | 0.97/轮 | 指数衰减 |
| 最小 ε | 0.015 / 0.02 | 两阶段探索概率下界 |
| 丢包率 | 2% | 模拟链路丢包 |
| 链路延迟 | 10ms | 基础传播延迟 |
| 抖动 | 3ms | 延迟随机抖动 |
| 验收种子 | 3、7、11 | 正常链路与带宽骤降分别评估 |

**训练过程统计：**

| 阶段 | 平均吞吐量(Mbps) | 平均RTT(ms) | 重传/轮 | 超时/轮 |
| --- | --- | --- | --- | --- |
| 正常链路 30 轮 | 0.587 | 63.2 | 11.8 | 3.4 |
| 带宽减半 30 轮 | 0.391 | 80.9 | 21.8 | 12.5 |

**多随机种子筛选结果（epsilon=0）：**

| 场景 | 模型 | 平均吞吐量(Mbps) | 平均RTT(ms) | 平均重传 | 平均超时 |
| --- | --- | --- | --- | --- | --- |
| 正常链路 | 旧 active | 0.592 | 61.0 | 10.0 | 2.0 |
| 正常链路 | 新策略（6 次） | 0.593 | 56.5 | 9.0 | 1.8 |
| 带宽减半 | 旧 active | 0.399 | 81.3 | 23.7 | 13.7 |
| 带宽减半 | 新策略（6 次） | 0.402 | 76.4 | 18.5 | 8.7 |

筛选不采用单次最好成绩，而是比较多个随机种子的均值和最差吞吐。新策略在正常链路吞吐不退化的同时降低 RTT；在带宽骤降场景中，平均重传减少约 21.8%，平均超时减少约 36.6%。完整数据保存在 `artifacts/training/q_mixed_v4_screening_summary.csv`。

**最佳 Q 表策略分析：**

以下为最终 active Q 表（`artifacts/models/active/q_table.json`）在各状态下的最优动作：

| 状态 | 最优动作 | Q 值 | 策略含义 |
| --- | --- | --- | --- |
| `rtt_up\|no_loss` | hold (0) | 26.26 | RTT 上升但不丢包，保持窗口避免拥塞加剧 |
| `rtt_up\|loss` | cwnd/2 (2) | 17.32 | RTT 上升且丢包，减半窗口 |
| `rtt_down\|no_loss` | cwnd+1 (1) | 26.22 | RTT 下降无丢包，继续探测带宽 |
| `rtt_down\|loss` | cwnd/2 (2) | 20.31 | 即使 RTT 下降，出现丢包仍减半窗口 |
| `rtt_stable\|no_loss` | cwnd+1 (1) | 23.07 | 网络稳定无丢包，逐步增加窗口 |
| `rtt_stable\|loss` | cwnd/2 (2) | 10.27 | 平稳但出现丢包，减半窗口响应拥塞 |

从策略可以看出，Q-Learning 控制器已经学到了拥塞控制的核心逻辑：RTT 上升时保持或减半，RTT 下降/稳定时逐步探测，出现丢包时果断减半。该策略与标准 TCP 拥塞控制的行为高度一致，但完全通过试错学习获得，无需人工设计规则。

项目提供 `demo_runner.py` 一键验证脚本，在统一链路条件下对比 AIMD 基线和 Q-Learning 控制器，并单独测试传输中途带宽减半。

**评估条件**：300 分组，2% 丢包率，10ms 基础延迟，3ms 抖动

## 9. 多线程控制

发送端包含三个并发执行流：

| 执行流 | 作用 |
| --- | --- |
| 主线程 | 按窗口大小发送新分组 |
| ACK 线程 | 调用 `recvfrom()` 接收 ACK 并更新确认状态 |
| Timer 线程 | 周期检查超时分组并重传 |

这些线程共享 `unacked`、`acked_packets`、`next_seq` 等状态，因此发送端使用 `threading.Lock` 对共享数据进行互斥访问，避免并发修改导致状态不一致。

## 10. 运行方式

启动接收端：

```bash
python3 receiver.py --host 127.0.0.1 --port 9001 --initial-seq 0
```

启动发送端：

```bash
python3 sender.py --target-host 127.0.0.1 --target-port 9001 --packets 40 --window-size 8 --rto 0.2
```

发送端运行结束后会输出确认分组数、重传次数、总耗时和平均吞吐量。

启用 Q-Learning 拥塞控制：

```bash
python3 sender.py --target-host 127.0.0.1 --target-port 9001 --packets 120 --window-size 8 --cc-mode qlearning --max-cwnd 32 --qtable-file artifacts/models/active/q_table.json
```

模拟丢包和延迟，便于观察窗口调整：

```bash
python3 receiver.py --host 127.0.0.1 --port 9001 --loss-rate 0.08 --delay-ms 20 --jitter-ms 10 --seed 1
```

多轮循环训练：

```bash
python3 train_q_learning.py --rounds 5 --packets 120 --q-table artifacts/models/active/q_table.json --q-epsilon 0.3 --epsilon-decay 0.85 --min-epsilon 0.05
```

## 11. 实验结果总结

### 11.1 可靠传输机制验证

当前版本完成了题目五中可靠传输底层架构的核心实现：Header 封装、累计 ACK、滑动窗口、RTO 定时重传、多线程控制、快速重传和乱序缓存，能够清楚展示如何在 UDP 之上用应用层代码构建可靠传输基础。所有场景下分组确认率达到 100%（300/300），验证了可靠传输机制的正确性。

### 11.2 Q-Learning 拥塞控制效果

Q-Learning 智能拥塞控制器通过 60 轮两阶段训练，学会了基于网络状态（RTT 趋势 + 丢包事件）自适应调整发送窗口。最终策略为：

- **任意 RTT 趋势 + 丢包** → 减半窗口
- **RTT 下降/稳定 + 无丢包** → 增加窗口
- **RTT 上升 + 无丢包** → 保持窗口

多随机种子筛选显示，新策略相对旧 active 在正常链路保持吞吐并降低 RTT，在带宽减半场景将平均重传从 23.7 降至 18.5、平均超时从 13.7 降至 8.7。最终 smoke test 中，Q-Learning 相对 AIMD 的结果如下：

| 指标 | Q-Learning vs AIMD |
| --- | --- |
| 吞吐量变化 | -0.2%（基本相同） |
| RTT 降低 | 47.7% |
| 重传减少 | 54.2% |
| 超时减少 | 71.4% |

因此，本次优化的主要收益不是追求某一次运行的最高吞吐，而是提高跨随机种子和突发拥塞场景下的稳定性。Q-Learning 的状态空间仅有 6 个离散状态，模型小、决策可解释，也便于继续训练和复现实验。
