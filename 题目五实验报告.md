## 摘要

本项目基于原生 UDP Socket，在应用层实现可靠传输的底层协议骨架。系统通过自定义 Header 封装分组序号、发送时间戳和固定长度数据；发送端维护未确认队列，并通过独立 Timer 线程执行 RTO 定时重传；接收端返回累计 ACK；ACK 线程根据 Timestamp 计算 RTT/SRTT。同时，系统实现了 3 个重复 ACK 触发快速重传和接收端乱序缓存。在此基础上，发送端使用 Q-Learning 控制拥塞窗口，以固定 100ms 控制周期内的 SRTT 趋势和丢包事件构造 6 个状态，并从保持、加 1、减半三个动作中选择下一步。最终 V9 模型采用 5% RTT 趋势阈值；RTT 上升但尚未丢包时先保持窗口，在 8 个随机种子下与 AIMD 公平筛选获得。

## 1. 实验目标

本实验聚焦题目五中的可靠传输底层架构，目标如下：

1. 在 UDP 之上自定义应用层数据包 Header。
2. 使用序号和累计 ACK 实现基础确认机制。
3. 发送端维护未确认队列，支持 RTO 超时重传。
4. 使用多线程拆分发送、ACK 接收和定时器扫描逻辑。
5. 设计 Q-Learning 智能拥塞控制器，根据网络反馈动态调整发送窗口。
6. 支持固定窗口与 Q-Learning 控制模式对比，便于观察智能控制效果。

## 3. 协议格式

### 3.1 数据包格式

| 字段 | 大小 | 含义 |
| --- | --- | --- |
| Sequence Number | 4 Bytes | 分组序号 |
| Timestamp | 8 Bytes | 发送端写入的 Unix 时间戳 |
| Payload | 1024 Bytes | 固定长度数据负载 |

在代码中，数据包格式由 `protocol.py` 中的常量定义：

```python
PAYLOAD_SIZE = 1024
DATA_HEADER_FORMAT = "!Id"
ACK_FORMAT = "!i"
```

其中 `!` 表示网络字节序，`I` 表示 4 字节无符号整数，`d` 表示 8 字节浮点数时间戳。

### 3.2 ACK 格式

ACK 报文只包含一个 4 字节字段：

| 字段 | 大小 | 含义 |
| --- | --- | --- |
| ACK Number | 4 Bytes | 当前累计确认到的最大连续序号，`-1` 表示尚无连续分组 |

## 4. Header 封装

`protocol.py` 提供四个核心函数：

- `pack_data_packet(seq, timestamp, payload)`：把序号、时间戳和负载封装为 UDP 可发送的字节串。
- `unpack_data_packet(packet)`：从收到的 UDP 数据中解析序号、时间戳和负载。
- `pack_ack(ack_number)`：封装 ACK。
- `unpack_ack(packet)`：解析 ACK。

发送端每次发送新分组或重传分组时，都会重新调用 `pack_data_packet()`，确保 Header 中的时间戳对应本次发送时间。

## 5. 可靠传输机制

### 5.1 发送端未确认队列

发送端使用 `unacked` 字典保存尚未收到 ACK 的分组。每个分组记录：

- `payload`：原始负载，用于重传。
- `last_send_monotonic`：最近一次发送时刻，用于计算是否超时。
- `wire_timestamp`：写入 Header 的 Unix 时间戳。
- `transmissions`：发送次数。

### 5.2 滑动窗口

发送端主线程按当前拥塞控制窗口控制在途分组数量。固定窗口模式下该值不变；Q-Learning 模式下该值由控制器每 100ms 更新一次：

```text
while next_seq < end_seq and len(unacked) < window_size:
    send_new_packet(next_seq)
```

这样可以避免一次性把所有 UDP 数据报发出，也为智能拥塞控制器提供直接的调节入口。

### 5.3 累计 ACK

接收端维护 `expected_seq`：

- 如果收到的 `seq == expected_seq`，说明分组按序到达，接收端推进 `expected_seq`。
- 如果收到更大的序号，说明出现乱序，接收端先缓存该序号。
- 每次处理完数据后，接收端返回 `expected_seq - 1` 作为累计 ACK。

发送端收到 ACK 后，会删除所有 `seq <= ack_number` 的未确认分组。

## 6. 定时重传

发送端启动独立 Timer 线程，每隔一小段时间扫描未确认队列：

```text
if now - last_send_monotonic >= RTO:
    retransmit(packet)
```

当检测到超时时，发送端会：

1. 使用原 Payload 重新封包。
2. 写入新的发送时间戳。
3. 更新最近发送时间。
4. 增加发送次数。
5. 重新通过 UDP Socket 发送。

由于 UDP 本身不保证可靠性，这一机制保证了丢失的分组最终可以再次发送。

## 7. 快速重传与乱序处理

基础 RTO 重传必须等待超时，恢复速度较慢。发送端 ACK 线程会统计重复累计 ACK：

```text
if duplicate_ack_count >= 3:
    retransmit(ack_number + 1)
```

当接收端先收到后续分组、但缺少当前期望序号时，会持续返回相同的累计 ACK。发送端连续收到 3 个重复 ACK 后，立即认为 `ack_number + 1` 对应分组可能丢失，并执行快速重传。

接收端使用 `buffered` 集合缓存乱序到达的序号。缺失分组到达后，接收端会连续推进 `expected_seq`，从而一次性确认已经缓存的后续分组。

## 8. Q-Learning 智能拥塞控制器设计

Q-Learning 控制器由 `sender.py` 内置实现，发送端通过 `--cc-mode qlearning` 启用。控制器不直接修改协议格式，而是每隔固定 100ms 统计网络反馈，再通过 Q-Table 决定下一周期的 CWND。旧实现使用动态 SRTT 作为周期长度，拥塞时 SRTT 变大会让一个周期收集更多 ACK，从而给拥塞状态带来不公平的正奖励；固定周期后，不同状态下的 ACK、重传和超时统计具有可比性。若一个周期既无 ACK 也无丢包事件，控制器跳过决策，避免空反馈触发无意义调窗。

### 8.1 状态设计

控制器严格按照题目要求，将最近一个 100ms 控制周期内的网络特征离散为 6 个状态：

| 状态维度 | 取值 | 含义 |
| --- | --- | --- |
| RTT 趋势 | `rtt_up` / `rtt_down` / `rtt_stable` | 当前 SRTT 相对上一控制周期上升、下降或保持稳定 |
| 丢包事件 | `loss` / `no_loss` | 本周期是否发生 RTO 或快速重传 |

因此 Q-Table 的状态集合为 `3 × 2 = 6` 个，例如 `rtt_up|loss` 表示 RTT 上升且本周期出现丢包，通常说明窗口过大或网络排队严重。V9 将趋势阈值 `rtt_trend_threshold_ratio=0.05` 写入 JSON：SRTT 相对变化超过 5% 即判定为上升或下降，否则归为稳定。最终搜索覆盖 3%、4%、5%、6%、8%、12%、16% 和 20%，5% 在吞吐与拥塞恢复之间取得最佳综合结果。

### 8.2 动作设计

动作集合为：

| 动作 | 窗口变化 |
| --- | --- |
| `0` | CWND 保持不变 |
| `1` | CWND + 1，主动探测更多带宽 |
| `2` | CWND / 2，用于响应拥塞信号 |

控制器不是每个 ACK 都立即调窗，而是在固定控制周期结束时，根据本周期统计得到的状态，通过 epsilon-greedy 策略选择下一周期动作。三个动作执行后都会将 CWND 保持为不小于 1 的整数，因而原始 Q-Learning CWND 曲线的端点是整数；图中的滑动平均趋势线可以是小数。

### 8.3 奖励函数

奖励函数采用复合形式，综合考虑吞吐量、RTT、重传和超时四个维度：

```text
R = throughput_weight × acked_packets
    - timeout_weight × timeout_events
    - retx_weight × retransmissions
    - rtt_weight × max(0, avg_rtt_ms - target_rtt_ms)
```

其中：
- `acked_packets`：本周期成功确认的分组数
- `avg_rtt_ms`：本周期平均 RTT（ms）
- `target_rtt_ms`：目标 RTT，仅惩罚超出目标的部分
- `retransmissions`：重传次数
- `timeout_events`：超时事件数

训练阶段通过上述奖励函数产生多个候选 Q 表，正式评估则使用 `--epsilon 0 --q-eval`，不探索、不在线更新。V9 以已训练候选为基础进一步筛选贪心策略和 RTT 趋势阈值，重点解决旧策略在 `rtt_up|no_loss` 状态直接减半导致窗口长期卡在 2--4、吞吐偏低的问题。

### 8.4 Q 值更新

每个有效控制周期结束时，控制器使用标准 Q-Learning / Bellman 公式更新上一个状态动作对：

```text
Q(s, a) = Q(s, a) + alpha * (reward + gamma * max Q(s', a') - Q(s, a))
```

程序参数 `--q-alpha` 控制学习率，`--q-gamma` 控制未来奖励折扣，`--epsilon` 控制探索概率。使用 `--qtable-file artifacts/models/active/q_table.json` 时，程序会在结束时保存当前 Q 表，下一次运行可以继续学习。历史权重备份保存在 `artifacts/models/backups/`。由于单次传输不一定收敛，项目提供 `train_q_learning.py` 进行多轮训练，并通过 `--epsilon-decay` 逐轮降低探索概率。

实现中还修复了首次 Bellman 更新污染问题。控制器启动时尚未执行任何动作，因此不存在可更新的上一状态动作对；当前代码将 `last_state` 和 `last_action` 初始化为 `None`，第一控制周期只选择动作，从第二周期开始才更新实际执行过的动作。这样可避免错误地修改 `rtt_up|no_loss → hold` 的初始 Q 值。

需要说明的是，Q-Learning 的 CWND 原始曲线不一定像固定规则控制器一样平滑。训练阶段存在 epsilon-greedy 随机探索，6 个离散状态会把不同网络情况映射到同一状态，动作也只有保持、加一和减半三类，因此窗口变化可能呈现抖动或阶梯状跳变。绘图时可以保留原始曲线，同时叠加滑动平均曲线，用原始曲线展示真实反馈，用平均曲线观察整体趋势。

### 8.5 多轮训练方法与训练结果

项目使用 `train_q_learning.py` 完成多轮训练，并使用 `screen_q_learning.py` 进行统一筛选。训练器每轮启动独立发送端和接收端，保存 metrics、history 和 checkpoint；筛选器让不同候选面对相同的分组数、链路参数和随机种子，同时测试正常链路与传输中途带宽减半两种场景。

最后一轮保持 6 个状态和 3 个动作不变，先比较 `rtt_up|no_loss` 下保持、加一和减半三种动作，再对最优的“先保持”策略搜索 3%--20% RTT 趋势阈值。筛选器使用相同的 300 分组、链路参数和 8 个随机种子测试正常链路与传输中途带宽减半两种场景。最终 5% 候选被安装为 V9 active 模型。

**最佳 Q 表策略分析：**

以下为最终 active Q 表（`artifacts/models/active/q_table.json`）在各状态下的最优动作：

| 状态 | 最优动作 | Q 值 | 策略含义 |
| --- | --- | --- | --- |
| `rtt_up\|no_loss` | hold (0) | 41.317 | RTT 上升但无丢包时先保持，避免过早减半损失吞吐 |
| `rtt_up\|loss` | cwnd/2 (2) | 26.945 | RTT 上升且丢包，立即减半窗口 |
| `rtt_down\|no_loss` | cwnd+1 (1) | 37.745 | RTT 下降且无丢包，继续探测带宽 |
| `rtt_down\|loss` | cwnd/2 (2) | 16.078 | 即使 RTT 下降，只要发生丢包仍减半 |
| `rtt_stable\|no_loss` | cwnd+1 (1) | 36.382 | 网络稳定无丢包，逐步增加窗口 |
| `rtt_stable\|loss` | cwnd/2 (2) | 17.121 | 稳定状态出现丢包时减半窗口 |

最终策略仍严格使用三个动作：RTT 下降或稳定且无丢包时加 1；任意状态出现丢包时减半；RTT 上升超过 5% 但尚未丢包时保持窗口。这个“先保持、见丢包再减半”的变化是 V9 吞吐接近 AIMD 的关键。

## 9. 多线程控制

发送端包含三个并发执行流：

| 执行流 | 作用 |
| --- | --- |
| 主线程 | 按窗口大小发送新分组 |
| ACK 线程 | 调用 `recvfrom()` 接收 ACK 并更新确认状态 |
| Timer 线程 | 周期检查超时分组并重传 |

这些线程共享 `unacked`、`acked_packets`、`next_seq` 等状态，因此发送端使用 `threading.Lock` 对共享数据进行互斥访问，避免并发修改导致状态不一致。

## 10. 运行方式

启动接收端：

```bash
python3 receiver.py --host 127.0.0.1 --port 9001 --initial-seq 0
```

启动发送端：

```bash
python3 sender.py --target-host 127.0.0.1 --target-port 9001 --packets 40 --window-size 8 --rto 0.2
```

发送端运行结束后会输出确认分组数、重传次数、总耗时和平均吞吐量。

使用候选 Q 表继续在线学习：

```bash
python3 sender.py --target-host 127.0.0.1 --target-port 9001 --packets 240 --window-size 1 --cc-mode qlearning --max-cwnd 64 --epsilon 0.1 --qtable-file artifacts/models/candidates/q_table_training.json
```

使用最终模型进行纯贪心评估，不更新或覆盖 Q 表：

```bash
python3 sender.py --target-host 127.0.0.1 --target-port 9001 --packets 300 --window-size 1 --cc-mode qlearning --max-cwnd 64 --epsilon 0 --q-eval --qtable-file artifacts/models/active/q_table.json
```

模拟丢包和延迟，便于观察窗口调整：

```bash
python3 receiver.py --host 127.0.0.1 --port 9001 --loss-rate 0.08 --delay-ms 20 --jitter-ms 10 --seed 1
```

多轮循环训练时建议写入候选文件，而不是直接覆盖 active：

```bash
python3 train_q_learning.py --rounds 18 --packets 240 --q-table artifacts/models/candidates/q_table_training.json --reset-q-table --q-alpha 0.025 --q-gamma 0.92 --q-epsilon 0.20 --epsilon-decay 0.94 --min-epsilon 0.03
```

多种子筛选：

```bash
python3 screen_q_learning.py --include-aimd --candidate v9=artifacts/models/active/q_table.json --seeds 3,7,11,13,17,23,29,31 --packets 300 --case-timeout-s 45
```

## 11. 实验结果总结

### 11.1 可靠传输机制验证

当前版本完成了题目五中可靠传输底层架构的核心实现：Header 封装、累计 ACK、滑动窗口、RTO 定时重传、多线程控制、快速重传和乱序缓存，能够清楚展示如何在 UDP 之上用应用层代码构建可靠传输基础。所有场景下分组确认率达到 100%（300/300），验证了可靠传输机制的正确性。

### 11.2 Q-Learning 拥塞控制效果

Q-Learning 智能拥塞控制器使用 6 个离散状态和 3 个动作。修复动态 SRTT 控制周期造成的奖励偏差后，控制器改为固定 100ms 周期，并通过策略比较、RTT 趋势阈值搜索和 8 个随机种子验证得到最终 V9 模型。其策略为：

- **RTT 下降/稳定且无丢包** → `cwnd + 1`
- **任意 RTT 趋势且发生丢包** → `cwnd / 2`
- **RTT 上升超过 5%，但无丢包** → `hold`

与同条件 AIMD 相比，V9 在正常链路只牺牲 0.8% 平均吞吐，RTT、重传和超时分别降低 58.9%、76.9% 和 78.6%；在带宽减半场景中，吞吐提高 9.0%，RTT、重传和超时分别降低 43.3%、69.8% 和 69.2%。

因此，V9 在保持可接受排队时延的同时，已经基本消除正常链路相对 AIMD 的吞吐差距。当前 Q-Table JSON 同时保存 Q 值、三个动作定义和 RTT 趋势阈值，模型体积小、决策可解释，也便于复现实验和回滚。
