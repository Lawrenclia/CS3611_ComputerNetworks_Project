# 题目五答辩讲稿提纲：可靠传输与 Q-Learning 拥塞控制版

## 1. 开场

大家好，我们负责的是题目五中“基于 UDP 的应用层可靠传输”这一部分。我的负责内容是 Q-Learning 智能拥塞控制器设计，它在原有可靠传输架构之上，以 1 个 RTT 为周期统计网络状态，并用 Q-Table 动态控制 CWND。

## 2. 实现目标

本部分主要实现三件事：

1. Header 封装：在 UDP 数据区中自定义序号、时间戳和固定负载。
2. 定时重传：发送端维护未确认队列，RTO 到期后自动重传。
3. 多线程控制：发送、ACK 接收和超时扫描分离运行。
4. 快速重传与乱序处理：连续 3 个重复 ACK 立即重传缺失包，接收端缓存乱序包。
5. 智能拥塞控制：使用 6 状态 Q-Learning 学习每个 RTT 周期后 CWND 应该保持、加 1 还是减半。

## 3. 系统结构

系统包含三个核心文件：

- `protocol.py`：负责数据包和 ACK 的二进制封装。
- `congestion_control.py`：负责固定窗口控制和 Q-Learning 智能拥塞控制。
- `sender.py`：负责可靠发送、ACK 处理和超时重传。
- `receiver.py`：负责接收数据、维护累计 ACK 并返回确认。

## 4. Header 设计

数据包由三部分组成：

- 4 字节 `Sequence Number`
- 8 字节 `Timestamp`
- 1024 字节 `Payload`

ACK 报文只包含 4 字节 `ACK Number`，表示当前已经连续收到的最大序号；如果还没有任何连续分组，则返回 `-1`。

## 5. 可靠传输流程

发送端主线程按当前拥塞窗口发送数据包，并把每个未确认分组放入 `unacked` 队列。固定窗口模式下窗口不变；Q-Learning 模式下窗口在每个 RTT 周期结束后更新。

接收端收到数据包后，根据序号判断是否按序到达，并返回累计 ACK。

发送端 ACK 线程收到累计 ACK 后，会删除所有 `seq <= ack_number` 的未确认分组。

## 6. 定时重传机制

发送端还有一个 Timer 线程，周期性检查所有未确认分组：

- 如果当前时间距离上次发送时间超过 RTO，就重新发送该分组。
- 重传时重新写入时间戳，并更新发送次数。

这样即使底层 UDP 丢包，也可以通过应用层逻辑恢复。

## 7. 快速重传与乱序处理

接收端如果先收到后续包，会把这些序号放进乱序缓存，并继续返回当前累计 ACK。

发送端 ACK 线程会统计相同 ACK 的次数，连续收到 3 个重复 ACK 后，立即重传 `ACK Number + 1` 对应的缺失包，不必等待 RTO 到期。

## 8. 多线程同步

发送端的主线程、ACK 线程和 Timer 线程都会访问未确认队列，所以程序使用 `threading.Lock` 做互斥控制，避免同时修改共享状态。

## 9. Q-Learning 拥塞控制设计

这一部分是我的负责内容。传统固定窗口的问题是窗口太小会浪费带宽，窗口太大又容易造成排队、丢包和重传。因此我把窗口控制抽象成一个强化学习问题：

- 状态：过去 1 个 RTT 内的 RTT 趋势和丢包事件，RTT 趋势分为变大、变小、平稳，丢包事件分为发生、未发生，因此一共 6 个状态。
- 动作：`0` 表示 CWND 保持，`1` 表示 CWND 加 1，`2` 表示 CWND 减半。
- 奖励：`R = alpha * 本轮成功吞吐量 - beta * 平均RTT - gamma * 丢包数量`。

每个 RTT 周期结束时，控制器根据当前状态用 epsilon-greedy 策略选择动作，并用 Bellman 公式更新上一次状态动作对。RTO 和快速重传不会绕过 Q-Table 直接降窗，而是计入本周期的丢包数量，进入状态和奖励函数。

代码运行时可以用：

```bash
python3 sender.py --cc q-learning --packets 120 --window-size 8 --min-window 1 --max-window 32 --q-table q_table.json
```

为了演示拥塞环境，也可以让接收端模拟丢包和延迟：

```bash
python3 receiver.py --loss-rate 0.08 --delay-ms 20 --jitter-ms 10
```

发送端日志中的 `[SENDER][CC]` 会显示状态、动作、窗口变化和奖励值，这部分可以作为答辩展示重点。

多轮训练可以用：

```bash
python3 train_q_learning.py --rounds 5 --packets 120 --q-table q_table.json --q-epsilon 0.3 --epsilon-decay 0.85
```

这个脚本会复用同一个 Q-Table，并让 epsilon 逐轮下降，用来体现“单次不一定收敛，需要多轮训练”的要求。

## 10. 总结

当前实现保留了可靠传输协议的基础骨架：自定义 Header、累计 ACK、滑动窗口、RTO 重传、多线程控制、快速重传和乱序缓存。在此基础上，Q-Learning 控制器把拥塞控制从固定参数改成根据网络反馈动态学习的策略，可以展示更完整的题目五设计思路。
