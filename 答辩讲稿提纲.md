# 题目五答辩讲稿提纲：可靠传输底层架构版

## 1. 开场

大家好，我们负责的是题目五中“基于 UDP 的应用层可靠传输”这一底层部分。当前版本只保留可靠传输架构。

## 2. 实现目标

本部分主要实现三件事：

1. Header 封装：在 UDP 数据区中自定义序号、时间戳和固定负载。
2. 定时重传：发送端维护未确认队列，RTO 到期后自动重传。
3. 多线程控制：发送、ACK 接收和超时扫描分离运行。
4. 快速重传与乱序处理：连续 3 个重复 ACK 立即重传缺失包，接收端缓存乱序包。

## 3. 系统结构

系统包含三个核心文件：

- `protocol.py`：负责数据包和 ACK 的二进制封装。
- `sender.py`：负责可靠发送、ACK 处理和超时重传。
- `receiver.py`：负责接收数据、维护累计 ACK 并返回确认。

## 4. Header 设计

数据包由三部分组成：

- 4 字节 `Sequence Number`
- 8 字节 `Timestamp`
- 1024 字节 `Payload`

ACK 报文只包含 4 字节 `ACK Number`，表示当前已经连续收到的最大序号；如果还没有任何连续分组，则返回 `-1`。

## 5. 可靠传输流程

发送端主线程按滑动窗口发送数据包，并把每个未确认分组放入 `unacked` 队列。

接收端收到数据包后，根据序号判断是否按序到达，并返回累计 ACK。

发送端 ACK 线程收到累计 ACK 后，会删除所有 `seq <= ack_number` 的未确认分组。

## 6. 定时重传机制

发送端还有一个 Timer 线程，周期性检查所有未确认分组：

- 如果当前时间距离上次发送时间超过 RTO，就重新发送该分组。
- 重传时重新写入时间戳，并更新发送次数。

这样即使底层 UDP 丢包，也可以通过应用层逻辑恢复。

## 7. 快速重传与乱序处理

接收端如果先收到后续包，会把这些序号放进乱序缓存，并继续返回当前累计 ACK。

发送端 ACK 线程会统计相同 ACK 的次数，连续收到 3 个重复 ACK 后，立即重传 `ACK Number + 1` 对应的缺失包，不必等待 RTO 到期。

## 8. 多线程同步

发送端的主线程、ACK 线程和 Timer 线程都会访问未确认队列，所以程序使用 `threading.Lock` 做互斥控制，避免同时修改共享状态。

## 9. 总结

当前实现保留了可靠传输协议的基础骨架：自定义 Header、累计 ACK、滑动窗口、RTO 重传、多线程控制、快速重传和乱序缓存。它证明了在 UDP 之上可以用应用层代码搭建可靠传输机制，也为后续继续扩展文件传输提供了基础。
